# Nexus AI Dashboard (工厂生产单流转看板) 系统设计与架构说明

## 1. 系统概述 (System Overview)

本系统名为 **Nexus AI Dashboard**（内部流转看板），主要用于工厂内部软包装生产订单的流转管理与跟踪。系统旨在解决传统纸质传单在不同工序（印刷、复膜、制袋、发货）之间的交接不透明、进度难追踪、异常难定位、以及数据易丢失的问题。通过全生命周期的流转看板管理，实现车间生产信息的实时同步和状态闭环。

## 2. 核心架构与业务流程 (Architecture & Workflow)

系统采用单页面应用(SPA)架构，基于 React + Vite + TypeScript 构建。主要流转实体为“生产工单”(Order/WorkOrder)。

### 2.1 整体订单生命周期 (Order Lifecycle)

系统中将生产工单的执行阶段分为如下标准工序：
1. **未起步 (Pending)**：新建的流转单尚未被任何工段接收。
2. **印刷 (Printing)**：第一道流转工序，需严格验证印膜尺寸和材料要求。
3. **复膜 (Lamination)**：第二道工序，处理多层复合结构（支持多层材料、厚度结构化展示）。
4. **制袋/分切 (Bag Making)**：成型工序，重点监控报废情况与袋形袋规。
5. **发货 (Shipping)**：物流入库及发货节点，包含检品确认。
6. **完成 (Completed)**：所有工序完结，订单历史归档。

### 2.2 状态机约束 (State Machine Rules)
- **单向流水线**：正常流转下，订单必须遵循 `接收 -> 执行 -> 完成登记 -> 下一环节`。
- **逆向流转（打回/回退）**：在任何环节如遇品质异常或者前道工序缺陷，可触发“回退至前环节”，被强制要求必填“回退原因(RollbackReason)”，保障问题追溯。

## 3. 核心功能模块设计 (Core Modules Design)

### 3.1 订单中心大盘 (Orders Dashboard)
**文件参考:** `src/components/Orders.tsx`

看板是系统最核心的页面，包含了完整的流水视图和强交互控制：
- **实时过滤层 (Filtration Layer)**：
  - **全局聚焦搜索 (Global Search)**：快捷定位订单、客户、袋型、备注、规格。当用户输入关键字进行搜索时，系统会自动将工作台视角定位至“全部”选项卡，确保用户可以跨工序查找结果，不丢失全局视野。
  - **状态页签切换**：可按不同流转工序切分单据集合进行查看（全部、印刷、复膜、制袋、发货、完成）。对于“全部”视图下，自动将已完成订单沉淀至底部。
  - **多维高级筛选 (Advanced Filters)**：支持基于“只看加急”、“只看异常”、特定关键备件（如压辊型号：65机型）、“开单时间窗”（今日/3天）、“滞留时长”进行过滤。支持针对开单时间或停留时间的多维升降排序。

- **订单卡片视图 (Order Card View)**：
  - **自适应视效设计**：大盘采用卡片流布局，每张卡片展示基础汇总信息。针对卡片折叠与展开的切换，做了信息关联继承，折叠时同样会利用副行展示该工单在当前工序的关键要求，确保概览下关键条件（如特定油墨、特定压辊）不消失。
  - **异常预警系统**：基于工序超时或滞留时长动态改变状态标(如红色高亮与呼吸脉冲动画)；具有异常错误字段纠错高亮卡片提示机制。

### 3.2 表单登记与交接控制 (Handover Context)
核心节点包含标准弹窗组件：
- **工序完工登记 (Completion Form)**：提供该工序结果量化上报，必须记录生产来源（机台/委外）、完成良品数量、单位与备注情况。
- **缺失项红警提示 (Validation Guard)**：支持数据“健康对齐检测”，对缺失参数做强烈的图形化红框提示与锚点定位，迫使操作层面上游或业务规范补全，保障下游不会受阻（例如：要求有薄膜厚度而遗漏）。

## 4. 技术栈选型与前端工程设计 (Tech Stack & Engineering)

### 4.1 技术栈基建
- **核心框架**: React 18, TypeScript (严谨化类型约束 `src/types`)
- **构建提效**: Vite
- **纯粹的样式引擎**: Tailwind CSS，实现原子化、可伸缩布局与主题体系。
- **图标与动效**: `lucide-react` (纯SVG矢量轻量图标) + `framer-motion` (实现柔性弹窗、层级交互与骨架抽屉动效)。

### 4.2 UI/UX 核心设计理念 (UX Philosophy)
1. **渐进式披露 (Progressive Disclosure)**: 
   在长列表大盘中，默认掩藏极度冗余的工艺层结构、历史流转轴；当用户关注并主动点击展开卡片时，这些模块以手风琴 `DetailSection` 以及格式化组件 `DataItem` 进行条理分明的透传。
2. **移动优先兼顾桌面 (Adaptive & Responsive)**: 
   全量采用 `md:` / `sm:` tailwind 前缀进行精细化的网格及字号配比策略。针对手机端限制的宽度，优化了横排字段展现、间距缩紧；宽屏端有更好的两列/三列表单和更丰富的边距呼吸感受。
3. **工业沉入式视觉架构 (Compact & Industrial Visual)**:
   - **紧凑专注**：针对顶部控制头、左侧边导航进一步做了空间优化（例如极简化的“系统运行正常”标志），力求在高度有限时提供尽量多的纵向空间给“工单列表”，减少滚动频次。
   - **工业感配色调号 (Color Coding)**: 选用深空灰(`slate-900`)作为沉稳主基调，搭配强功能性色彩：深紫(`purple/复膜`)、靛青(`indigo/印刷`)、翠绿(`emerald/发货`)、琥珀(`amber/警告`)等。

## 5. 组件级工程抽象 (Component Patterns)

### 大单据卡片 `OrderCard`
由于传统ERP订单复杂度极高，组件避免了堆叠乱写，将其内部划分为多块 `ProcessModule`：
```tsx
// 概念示例
<OrderCard>
  <Header 承载基础元数据(款号/加急度) />
  {isExpanded && (
     <ProcessModule title="印刷工序" icon={...}> ... </ProcessModule>
     <ProcessModule title="覆膜工序" icon={...}> <LayerTable /> </ProcessModule>
     <ProcessModule title="制袋工序" icon={...}> ... </ProcessModule>
  )}
  <ActionButtonsBottomBar />
</OrderCard>
```
辅以通用的 `<DataItem>` 函数式组件，保证所有业务参数(Key-Value)标签字体字重样式绝对统一，避免样式碎片化。

## 6. 后续演进路线建议 (Future Roadmap)

1. **真实服务端接驳 (Backend Overhaul)**：
   目前的 `mock` 状态最终需要对接生产级别的 RESTFul API / GraphQL，并建议引入 `WebSocket`/`SSE` 将车间一线的报警和工序变化作为实时流广播推送到所有面板终端（无需刷新页面即可看见队列推进）。
2. **设备 IoT 联网打通 (Machine IoT Sync)**：
   工序登记不仅依赖人工“完成登记”，可扩展接入工业物联网(PLC接口)，通过数据采集器自动识别卷膜下机、设备产出米数，实现真正无人化或半自动化报工。
3. **基于角色的深层访问权限 (RBAC Implementation)**：
   对于不同终端（对应不同生产车间小组），隐藏无关面板；并且硬性校验对应角色的完工审核、打回流转权限，防范越级错改车间产量。
4. **报表与BI模块扩容 (BI & Ops Center)**：
   利用完工单与回退原因历史（如：耗材废品率、退改时间轴）计算出月度交付履约率(OTD)，支撑管理者做精准归因提效指导。
